# Модуль: Конструктор ботов и сценариев

No-code конструктор ветвящихся диалоговых сценариев WhatsApp-бота: администратор создает и меняет логику диалогов без привлечения разработчиков. Источник истины по бизнес-логике — [ТЗ, Модуль 1](../technical-specification.md); сводка инвариантов — [business-rules.md](../business-rules.md).

## Сценарии и триггеры

Сценарий бота — не история сообщений, а схема поведения: что запускает диалог → что отправить → какой ответ ждать → что сделать дальше. Журнал входящих и исходящих сообщений ведётся отдельно.

Вместо одного огромного графа у бота несколько отдельных сценариев, каждый со своим триггером:

| Триггер | Что запускает | Сколько сценариев |
| --- | --- | --- |
| Входящее сообщение | Главный диалог: человек сам пишет боту. | Один (главный). |
| Новая заявка | Автосценарий: заказчик выбрал объявление — уведомить поставщика. | Работает самый старый опубликованный; публикация «затенённого» дубля даёт предупреждение. |
| Истекает объявление | Автосценарий: за сутки до конца публикации — опрос актуальности. | Аналогично. |

Главный диалог ведёт пользователя через сессию основного разговора (см. «Исполнение главного сценария»). Сценарии остальных триггеров — **run-based**: каждое событие создаёт отдельный, изолированный запуск сценария.

## Запуски (runs) и маршрутизация ответов

- Каждая проактивная отправка (уведомление о заявке, опрос актуальности) — отдельный запуск сценария. Запуск не перезаписывает основной диалог контакта: у одного поставщика могут одновременно ждать ответа несколько заявок и объявлений, плюс идти обычный разговор с ботом.
- Каждая кнопка сообщения запуска несёт уникальный непрозрачный токен запуска. Ответ сопоставляется только по токену — никогда по тексту кнопки — и попадает точно в тот запуск, версию и блок, которые отправили сообщение, на каком бы шаге главного диалога ни находился контакт.
- Запуск закрепляется за опубликованной версией сценария на момент отправки. Республикация не перекраивает уже отправленные сообщения: кнопка, нажатая через несколько дней, ведёт по той ветке, которая её отправила.
- Клик по кнопке уже завершённого запуска получает вежливое «вопрос уже закрыт»; чужой или устаревший токен молча игнорируется.
- Запуск, чьё первое сообщение не удалось доставить (закрытое окно без утверждённого шаблона, ошибка транспорта), помечается ошибкой и виден в журнале запусков; инициатор (например, ежедневный цикл актуальности) может повторить попытку позже.
- Кнопки, отправленные до перевода флоу на сценарии, продолжают обрабатываться по-старому — ответы на них не теряются.

## Визуальный редактор схем

Сценарий создается и редактируется как схема (граф) на холсте: блоки-шаги соединяются линиями переходов, и администратор видит всю логику диалога целиком. Визуальный эталон — конструктор [watbot.kz](https://watbot.kz/images/slider/1.webp).

Редактор доступен в админ-панели на странице «Сценарии» раздела «Бот» (`/admin/bot-scenario`); главный сценарий с блоком «Старт» создается автоматически при первом открытии. Вверху страницы — переключатель сценариев и создание нового (название + триггер); удалить можно только ни разу не публиковавшийся сценарий — у опубликованного могут ждать ответа уже отправленные сообщения.

Возможности холста:

- добавление блока кнопкой на панели инструментов (набор блоков зависит от триггера сценария), перетаскивание блоков по холсту;
- создание связи: клик по кружку выхода, затем клик по блоку-цели (Esc — отмена), либо протягивание линии от кружка к блоку; повторное соединение с того же выхода переназначает связь, протягивание в пустое место — разрывает (у одного выхода — не больше одной связи); подсказка о способе соединения показывается над холстом;
- альтернатива без мыши: в боковой панели выбранного блока раздел «Переходы» — выпадающий список «куда ведёт» по каждому выходу (включая «Любую другую фразу»); блоки в списках подписаны содержимым (текст, условие, действие или текст шаблона), а одинаковые различаются номером — «Условие: Заявка ещё ожидает ответа № 2»;
- редактирование выбранного блока в боковой панели: текст сообщения, варианты ответа (с лимитами WhatsApp), канал и шаблон у блока «WhatsApp-сообщение», условие/действие у соответствующих блоков; блок «Старт» не удаляется;
- масштабирование (колесо/кнопки, 30–175 %), центрирование всей схемы, поиск по тексту и вариантам блоков;
- сохранение черновика и публикация — отдельными кнопками (см. «Публикация»).

## Типы блоков

### Блоки главного диалога

| Блок | Поведение | Выходы |
| --- | --- | --- |
| Старт | Точка входа. Единственный в сценарии, создается автоматически, не удаляется. | «Продолжить» |
| Текст | Отправляет сообщение и сразу переходит дальше. | «Продолжить» |
| Интерактивное меню (кнопки) | Сообщение с кнопками, до 3 (лимит WhatsApp). | Выход на каждую кнопку + «Любая другая фраза» |
| Интерактивный список | Сообщение со списком, до 10 элементов (лимит WhatsApp). | Выход на каждый элемент + «Любая другая фраза» |
| Запрос ввода (AI) | Передает управление AI-ассистенту: сбор объявления поставщика или поиск для заказчика — задача настраивается у блока (см. [ai-assistant.md](ai-assistant.md)). | «Продолжить» — после завершения работы AI |
| Мои объявления (CTA) | Отправляет интерактивное сообщение с URL-кнопкой «Открыть кабинет» и персональной подписанной ссылкой в веб-кабинет поставщика; текст сообщения настраивается. | «Продолжить» — сразу после отправки |
| Завершение | Явно заканчивает диалог. | — |

Если вариантов ответа больше трех, вместо кнопок используется интерактивный список.

### Блоки run-based сценариев

Интерактивные меню, списки и AI-блоки в run-based сценариях недоступны (и наоборот, блоки ниже недоступны в главном диалоге, кроме общих «Текст», «Мои объявления (CTA)» и «Завершение»):

| Блок | Поведение | Выходы |
| --- | --- | --- |
| WhatsApp-сообщение | Проактивное сообщение с учётом 24-часового окна. Канал: **адаптивно**, **только сессионное** или **только шаблон**. Для каналов с шаблоном текст сообщения задаёт **тело шаблона Meta** — единый источник: вне окна уходит платный шаблон (только утверждённый), в открытое окно тот же текст с подставленными `{{n}}` уходит бесплатным сессионным сообщением (утверждение Meta для него не требуется — работает и пока шаблон на модерации). Своё поле текста есть только у канала «только сессионное». У блока настраиваются: шаблон из реестра, сопоставление переменных `{{n}}` с данными заявки/объявления/контакта, до 3 кнопок (у шаблона должно быть столько же кнопок быстрого ответа) и таймаут ожидания ответа в часах. Без кнопок — отправил и пошёл дальше. | Выход на каждую кнопку; «Таймаут» — если задан срок; «Продолжить» — если кнопок нет |
| Условие | Ветвление по текущему состоянию домена: «заявка ещё ожидает ответа», «объявление опубликовано», «окно WhatsApp открыто». Проверяется в момент прохождения блока — в том числе когда ответ пришёл спустя дни. | «Да» / «Нет» (оба обязательны) |
| Действие | Доменное действие: принять/отклонить заявку, продлить/архивировать объявление, отправить CTA-ссылку на кабинет, уведомить заказчика об исходе заявки. Жёсткие инварианты остаются в домене: действие с нарушенным предусловием (например, повторное решение по заявке) тихо пропускается — ветвиться на такие случаи должен блок «Условие». | «Продолжить» |
| Завершение | Заканчивает запуск. | — |

В сессионных текстах run-based сценариев (блок «Текст», блок «WhatsApp-сообщение» с каналом «только сессионное») можно использовать именованные подстановки данных — например `{{listing.category}}`, `{{request.query}}`, `{{contact.name}}`; доступный набор зависит от триггера. Для каналов с шаблоном текст берётся из тела шаблона, а переменные сопоставляются по номерам `{{n}}` в настройках блока: строки сопоставления создаются автоматически по числу `{{n}}` выбранного шаблона (оператор только выбирает значения; если шаблона нет в реестре, сохранённое сопоставление не трогается). Изменить формулировку можно только новым шаблоном в реестре (Meta не разрешает редактировать существующие).

### Таймаут ожидания

Если у блока «WhatsApp-сообщение» задан таймаут и ответ не пришёл в срок, запуск идёт по выходу «Таймаут» (пример ветки: «нет ответа 24 часа → архивировать»). Неподключённый выход «Таймаут» тихо завершает запуск. Таймауты проверяются автоматикой раз в час; ответ, успевший раньше проверки, отменяет таймаут.

## Выходы и связи

- Каждый выход блока соединяется ровно с одним блоком; на вход блока может приходить любое число связей.
- Выбор кнопки или элемента списка переводит пользователя по соответствующему выходу. В главном диалоге текстовый ответ, совпадающий с названием кнопки/элемента, приравнивается к его выбору; в run-based сценариях ответы сопоставляются только по токену кнопки.
- «Любая другая фраза» — служебный выход интерактивных блоков главного диалога: сюда попадает любой ввод, не совпавший ни с одним вариантом. Если выход не подключен, бот повторяет сообщение текущего шага.
- «Продолжить» — переход по умолчанию: у текстовых блоков срабатывает сразу после отправки сообщения; у блока «Запрос ввода (AI)» — после того, как AI завершил сбор данных, в том числе по исчерпании лимита уточнений (см. [business-rules.md](../business-rules.md)).

## Публикация и мягкое обновление активных сессий

- Правки схемы не влияют на пользователей, пока сценарий не опубликован: администратор редактирует черновик схемы и применяет его кнопкой «Опубликовать сценарий». Публикация также сохраняет текущий черновик.
- Каждая публикация фиксирует неизменяемый снимок версии — уже идущие запуски продолжают следовать своим снимкам (см. «Запуски»).
- Публикация блокируется при ошибках валидации: превышены лимиты WhatsApp (больше 3 кнопок / 10 элементов списка, название кнопки длиннее 20 символов, элемента списка — 24), не подключен выход блока «Старт», у интерактивного блока или блока «WhatsApp-сообщение» есть неподключенные выходы кнопок, у блока «Условие» не подключены «Да»/«Нет» или не выбрано условие, у блока нет текста сообщения (для «WhatsApp-сообщение» — только при канале «только сессионное»: остальным текст задаёт шаблон), блок недоступен для триггера сценария, не выбран шаблон при канале с шаблоном, число переменных не совпадает с шаблоном, число кнопок не совпадает с шаблоном (проверяется, только когда состав кнопок шаблона известен реестру — синхронизация из Meta может зеркалить шаблон без него; несовпадение, не пойманное валидацией, всплывёт ошибкой отправки в журнале запусков), подключён выход «Таймаут» без срока (или срок без кнопок), связь ведет на несуществующий блок, с одного выхода идёт больше одной связи. Ошибки показываются списком в уведомлении. Блоки, недостижимые от «Старта», и не найденный/не утверждённый в реестре шаблон — предупреждения, публикацию не блокируют (вне окна такой шаблон всё равно не отправится — запуск упадёт с ошибкой и виден в журнале).
- Выход «Любая другая фраза» подключать не обязательно — без связи бот повторяет текущий шаг.
- При публикации активные сессии главного диалога не обрываются жестко: пользователь завершает текущий шаг и переводится на актуальную ветку; при критическом конфликте узлов — мягкий сброс в главное меню. Критический конфликт — текущий шаг удалён, перестал ожидать ввода либо изменился по существу: сменился тип блока, набор или названия кнопок/элементов, задача AI-блока. Правка только текста сообщения конфликтом не считается — шаг пользователя сохраняется.

## Исполнение главного сценария (рантайм)

- Пользователей ведет только опубликованная схема; пока сценарий не опубликован, бот не отвечает на входящие.
- Прежде чем сообщение попадает в главный диалог, оно проверяется на ответ по кнопке запуска (по токену) — такие ответы обрабатываются своим запуском и в диалог не попадают.
- **Новый диалог** начинается с блока «Старт», когда: пользователь пишет впервые; предыдущий диалог дошел до конца ветки; с последнего взаимодействия прошло более 24 часов (зеркально 24-часовому окну WhatsApp).
- Получив сообщение, бот проходит цепочку блоков подряд: текстовые блоки отправляются один за другим по выходу «Продолжить», пока диалог не остановится на блоке, ожидающем ввода (меню, список, AI), или ветка не закончится.
- На интерактивном блоке нажатие кнопки/выбор элемента ведет по соответствующему выходу; текст, совпадающий с названием варианта (без учета регистра и краевых пробелов), приравнивается к выбору; прочий ввод уходит по «Любой другой фразе», а если она не подключена — бот повторяет сообщение текущего шага.
- Сообщения одного пользователя обрабатываются строго по одному, чтобы параллельные ответы не путали позицию в диалоге.
- Защита от зацикливания: за одно входящее сообщение бот проходит не более 20 блоков подряд; при превышении диалог завершается (следующее сообщение начнет новый). Тот же потолок действует на автопроход блоков внутри запуска.
- Блок «Запрос ввода (AI)» передает ход AI-ассистенту и удерживает пользователя на себе, пока AI не завершит сбор данных (см. [ai-assistant.md](ai-assistant.md)).

## Страницы админки

- **«Сценарии»** (`/admin/bot-scenario`) — переключатель сценариев и визуальный редактор.
- **«Запуски»** (`/admin/scenario-runs`) — журнал запусков: сценарий, версия, контакт, предмет (заявка/объявление), статус (ждёт ответа / завершён / ошибка отправки), ожидающий блок и таймаут; фильтры по статусу и сценарию.
- Реестр и модерация Meta-шаблонов остаются отдельной страницей «Шаблоны WhatsApp» — в сценарии оператор только выбирает шаблон из реестра.

## Типовые сценарии

Команда `bot:install-default-scenario` устанавливает и публикует три типовых сценария: главный диалог и два флоу — «Новая заявка», «Продление объявления» (см. [user-flows.md](user-flows.md), [listings-lifecycle.md](listings-lifecycle.md)). Уже опубликованный сценарий перезаписывается только с `--force`; каждый сценарий оценивается отдельно. Пока сценарий события не опубликован, соответствующее уведомление работает по прежним встроенным правилам — поведение для пользователя то же.
